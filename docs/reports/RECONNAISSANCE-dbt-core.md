## Repository Composition

**Languages:** Python (core execution engine, CLI, parsers, graph algorithms), YAML (`.changes/`, issue templates, CI workflows, dbt project fixtures), SQL (model fixtures used in functional tests), TOML (build configuration).

**Size:** ~796 Python files across `core/dbt/`, `tests/`, and supporting scripts. A genuine production framework codebase.

**Why chosen:** dbt-core is the canonical data transformation framework at the heart of the modern data stack. It uses a DAG-based execution model, has a rich YAML + SQL + Python polyglot structure, and its architecture (parser → manifest → graph → executor) mirrors the kind of pipeline topology the Cartographer is designed to map. It is also explicitly called out as a primary target in the TRP spec.

## The Five FDE Day-One Questions — Manual Answers

### 1. What is the primary data ingestion path?

Data enters dbt-core as a **project configuration**: `dbt_project.yml` (resolved by `core/dbt/cli/resolvers.py`) defines the project structure; `profiles.yml` defines the target database connection. The parse phase then reads YAML source definitions (`core/dbt/parser/sources.py`), SQL model files (parsed by `core/dbt/parser/models.py`), and schema tests (via `core/dbt/parser/schema_renderer.py`). These are rendered with Jinja context (`core/dbt/config/renderer.py` + `core/dbt/context/target.py`) and assembled into a `Manifest` object (`core/dbt/contracts/graph/manifest.py`). The manifest is the internal representation of the entire project graph.

**Summary:** `dbt_project.yml` → CLI resolvers → Parser pipeline → Manifest.

### 2. What are the 3–5 most critical output datasets/endpoints?

1. **Manifest artifact** (`manifest.json`) — serialized project graph uploaded to dbt Cloud and consumed by downstream BI tools; generated by `core/dbt/parser/manifest.py`
2. **Run results** (`run_results.json`) — per-model execution outcomes (success/fail, compiled SQL, timing); consumed by monitoring tooling
3. **Compiled SQL models** (written to `target/compiled/`) — the SQL dbt generates after Jinja rendering; the actual DDL/DML executed against the warehouse
4. **Structured event logs** — telemetry events sent via `core/dbt/tracking.py` to Snowplow (anonymous usage) and structured logs via `core/dbt/events/types.py`
5. **CLI exit codes** — wrapped by `core/dbt/cli/exceptions.py` (`ResultExit`, `ExceptionExit`); consumed by CI/CD pipelines to determine pass/fail

### 3. What is the blast radius if the most critical module fails?

`core/dbt/node_types.py` is the architectural keystone. It defines `NodeType` enumerations and category lists (`REFABLE_NODE_TYPES`, `EXECUTABLE_NODE_TYPES`, etc.) that every other subsystem consults to classify and route nodes. A breaking change here would fail:
- `core/dbt/graph/graph.py` — DAG construction and traversal
- `core/dbt/graph/selector_methods.py` — all node selection logic (by path, tag, FQN, state)
- `core/dbt/parser/common.py` — node type validation during parse
- `core/dbt/task/` — any task that filters nodes before execution

`core/dbt/exceptions.py` (second-highest PageRank) is the global error contract. A break there disables graceful failure modes system-wide.

### 4. Where is the business logic concentrated vs. distributed?

**Concentrated:**
- **Parsing pipeline** (`core/dbt/parser/`) — YAML-to-object conversion, schema rendering, source resolution: complex, interdependent, hard to change without cascading test failures
- **Graph algorithms** (`core/dbt/graph/`) — DAG construction, topological ordering, node selection; the correctness core of the product
- **Configuration management** (`core/dbt/contracts/project.py`, `core/dbt/config/renderer.py`) — Jinja rendering with secret masking, profile merging

**Distributed:**
- **Node execution** — adapter-specific materialization logic lives in adapter packages (`dbt-postgres`, `dbt-bigquery`, etc.), not in dbt-core; this repo only defines the interface (`core/dbt/adapters/protocol.py`)
- **Error handling** — domain-specific exceptions scattered: `core/dbt/exceptions.py` (core), `core/dbt/cli/exceptions.py` (CLI), `core/dbt/artifacts/exceptions/` (artifact validation)
- **Test fixtures** — functional test infrastructure is distributed across `tests/functional/` with hundreds of sub-packages, each simulating a mini dbt project

**Key observation:** The parser and graph layers are intentionally centralized; the adapter execution layer is intentionally distributed — this separation is the product's main architectural decision.

### 5. What has changed most frequently in the last 90 days?

Based on running `git log --since="90 days ago" --name-only`:
- `tests/functional/semantic_models/test_semantic_model_v2_parsing.py` — 6 commits; semantic layer v2 is actively being built
- `tests/functional/semantic_models/fixtures.py` — 5 commits; test support for the same feature
- `core/dbt/compilation.py` — 4 commits; core compilation pipeline being refactored
- `core/dbt/cli/params.py` — 3 commits; new CLI flags being added
- `core/dbt/events/types.py` — 3 commits; structured logging types updated

**Pattern:** The **semantic layer v2** (`semantic_models/`) is the primary area of new feature development. The compilation and parser subsystems are being refactored in parallel. CI/CD and GitHub workflows have also been updated frequently, suggesting infrastructure investment.

## Difficulty Analysis

**Time constraint:** The `core/dbt/` directory has ~400 Python files with deep package nesting (`core/dbt/artifacts/resources/v1/`, `core/dbt/context/`, `core/dbt/graph/`, etc.). 30 minutes was insufficient to trace the full parse-to-execution path.

**Hardest aspect:** Understanding where the adapter boundary sits. The repo contains adapter *protocols* and base classes but no concrete adapter — you keep following imports that resolve to `NotImplementedError` until you realize the execution layer is split across separate packages.

**Where I got lost:** The Jinja context construction chain. `core/dbt/context/providers.py`, `configured.py`, `target.py`, `docs.py`, and `macros.py` form a class hierarchy where each level adds context variables — hard to reconstruct mentally without running the code.

**Key insight for architecture priorities:** The Hydrologist's value here is not the traditional table-level lineage (dbt models *are* the lineage definition). Instead, the Surveyor's PageRank is the primary signal: high-centrality modules like `node_types.py` and `exceptions.py` tell the story faster than the lineage graph for this type of framework repo.
